# 개발 모드

> 빠르게 구현하되, 시스템의 안정성과 문서 체계를 깨지 않는다.

## 설계 철학

- AI는 설계 없이 달린다 → **최소한의 사전 분석** 강제
- AI는 자신감 있게 틀린다 → **자기 검수** 내장
- 기존 구조를 존중하되 과도한 설명은 하지 않는다

---

## 실행 절차

### 1. 요구사항 재정의 (2~3줄)

사용자의 요청을 기술적으로 재정의한다. 해석이 여러 개이면 나열하고 확인한다.

```
목표: {무엇을 달성하는가}
범위: {어떤 파일/모듈이 영향받는가}
제약: {건드리면 안 되는 것}
```

### 2. 영향도 간단 분석

기존 코드를 읽고 아래를 파악한다:

- 변경 대상 파일 목록
- 기존 패턴/컨벤션 확인
- 호출자/의존 관계 (1-depth)

> 장황한 설계 설명 금지. 분석은 코드 변경에 필요한 최소 범위만.

### 3. 최소 변경 구현

**원칙**:
- 요청받은 것만 변경한다
- 인접 코드 "개선" 금지
- 기존 스타일에 맞춘다
- 변경된 모든 줄이 사용자의 요청에 직접 연결되어야 한다

**구조 분리 & 캡슐화**:
- 다른 기능/화면에 대한 변경은 파일(또는 클래스) 단위로 분리한다
- 단일 PR에서 여러 화면을 건드릴 때, 각 화면별 변경 범위를 명확히 분리한다 (파일/패키지/커밋 단위)
- 하나의 메서드에 여러 기능을 몰아넣지 않는다. "준비 → 실행 → 후처리"가 섞이면 각 단계별 메서드로 쪼개고, 외부에는 의도가 드러나는 API만 노출한다
- 사용자에게 보이는 문자열은 항상 strings.xml로 분리한다 (예외: 로그 태그/디버그 전용 임시 문구)
- 복잡도가 높아지는 로직(분기/상태/경계조건)에는 메서드 단위로 "무엇/왜" 중심의 짧은 주석을 추가한다. 코드 중복 설명이 아니라 의사결정/제약/트레이드오프를 기록한다

**예외 규칙 (생산성 보호)**:
- 10줄 이하의 단순 핫픽스에는 과도한 파일 분리/리팩토링을 하지 않는다 (최소 변경 우선)
- 단, 반복될 가능성이 높은 로직이거나 버그 원인 가능성이 높은 결합이면 바로 분리한다
- 주석은 과도하게 달지 않는다. "나중에 까먹기 쉬운 의도"만 남긴다

### 4. Self Review

구현 완료 후 변경된 코드를 다시 읽고 아래를 점검한다:

| 점검 항목 | 확인 |
|----------|------|
| null/예외 안전성 | NPE 가능성, 빈 컬렉션 처리 |
| 책임 분리 위반 | 변경이 해당 레이어에 적절한가 |
| 누락된 호출자 | 시그니처 변경 시 모든 호출처 업데이트 |
| 컴파일 | 빌드 성공 확인 |

문제 발견 시 **즉시 수정** 후 보고.

### 5. 결과 보고

```
✅ 완료
- 변경 범위: {어떤 파일/어떤 책임}
- 핵심 변경: {한 줄 요약}
- strings.xml: {추가/변경 여부}
- 주석 포인트: {복잡한 경우, 메서드별 주석 추가 위치}
- 테스트 포인트: {확인이 필요한 시나리오 1~2개}
```

---

## 하지 않는 것

- ❌ 대규모 리팩토링 (필요하면 옵션으로만 언급)
- ❌ 요청 외 기능 추가
- ❌ 설계 장황 설명
- ❌ 불가능한 시나리오에 대한 방어 코드
- ❌ "더 나은 구조" 자발적 제안 (물어보면 답변)
