# 계획 모드

> AI가 바로 코드로 뛰어들지 못하게 "브레이크"를 거는 모드.

## 설계 철학

- AI는 설계 없이 달린다 → **구현 전 설계 강제**
- AI는 기억을 못한다 → **외부 문서 기반으로 사고**
- 구조 변경은 합의 없이 진행하지 않는다

---

## 절대 규칙

**이 모드에서 코드를 작성하지 않는다.**

설계와 합의만 수행한다. 코드는 `/개발` 모드에서.

---

## 실행 절차

### 1. 요구사항 명확화

```
목표: {최종 달성 상태}
배경: {왜 필요한가}
제약: {바꾸면 안 되는 것, 호환성 요구}
불명확: {해석이 갈리는 부분 → 질문}
```

불명확한 점이 있으면 **가정하지 않고 질문**한다.

### 2. 현재 구조 파악

관련 코드를 읽고 아래를 정리한다:

- 관련 파일/클래스 목록
- 현재 데이터 흐름
- 기존 패턴/컨벤션

> 프로젝트에 `docs/AI_CONTEXT.md`, `docs/ARCHITECTURE.md`가 있으면 먼저 읽는다.

### 3. 설계 구조 제시

```
[변경 대상]
├── 파일 A — {역할} — {변경 내용 요약}
├── 파일 B — {역할} — {변경 내용 요약}
└── 파일 C (신규) — {역할}
```

### 4. 데이터 흐름 정의

변경 전후의 데이터 흐름을 명시한다:

```
[Before]
A → B → C

[After]
A → B → D (신규) → C
```

### 5. 상태 소유자 정의

상태가 관련되면 소유자를 명확히 한다:

| 상태 | 소유자 | 변경 주체 | 구독자 |
|------|--------|----------|--------|
| {상태명} | {ViewModel/Repository 등} | {누가 변경} | {누가 읽음} |

### 6. 리스크 분석

최소 **3개 이상** 리스크를 제시한다:

| # | 리스크 | 발생 조건 | 대응 |
|---|--------|----------|------|
| 1 | | | |
| 2 | | | |
| 3 | | | |

### 7. 구현 전 합의

사용자에게 아래를 확인받는다:

```
확인 사항:
1. 설계 방향이 맞는가?
2. 리스크 대응이 적절한가?
3. 빠진 요구사항이 있는가?
4. 진행해도 되는가?
```

**합의 없이 구현으로 넘어가지 않는다.**

## /계획 → /개발 전환

사용자가 합의하면:

```
✅ 계획 승인됨. /개발 모드로 구현을 시작합니다.
```

이후 `/개발` 모드의 절차를 따른다.
